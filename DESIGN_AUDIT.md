# Aravosh — Design Audit

## Product & audience interpretation

Aravosh is an early-stage, founder-led technology studio (per its own copy: "we do
not pretend to have a long agency track record"). Site content and the site owner's
own stated priorities (confirmed in project discussion, not inferred) rank the
business in this order:

1. AI agents, multi-agent systems, agent orchestration
2. Internal tooling / workflow integrations
3. Web development

The realistic buyer is a technical decision-maker (founder, CTO, eng lead) at a
small-to-mid US company evaluating a vendor for a scoped AI/software project. This
is a skeptical, detail-reading audience, not a consumer-marketing audience — the
existing copy already leans into that ("no fixed price list," "we do not pretend,"
"real case studies should be specific"). The visual design has not caught up to that
honest, engineering-grade voice; it still wears a generic "AI startup" costume from
an earlier iteration of the project.

## Stack

Static HTML generated by `build.mjs` (a hand-rolled template generator, no
framework) → 9 committed `.html` files, one shared `styles.css` (1249 lines) and
`script.js` (561 lines). No package.json, no bundler, no lint/test config. The only
"build" is `node build.mjs`, which regenerates the HTML from the templates in
`build.mjs`; the generated files are committed and served by Vercel with no
production build step. There is no automated test/lint/typecheck suite to run as a
baseline — this is worth noting honestly rather than fabricating tool output.

## Current visual language

A dark-theme "AI startup" aesthetic that was partially, not fully, migrated to a
light theme (see `git log`: "Repair light theme site layout"). The result is two
themes' worth of decoration layered on top of each other:

- An animated gradient "aurora" blob behind the hero, a radial `hero-glow`, a faint
  grid-pattern background, and a cursor-following radial "spotlight."
- A 3D CSS scene in the hero: glassmorphic floating cards with backdrop-blur, two
  rotating "orbit" rings, three glowing "spark" particles, and a full canvas-based
  animated particle sphere (`orb-canvas`, ~80 lines of JS driving a hand-rolled 3D
  point-cloud renderer) — entirely decorative, not a product screenshot or diagram.
- Animated gradient-fill "shimmer" text on the hero H1 and the 404 code.
- A cursor-following "magnetic" offset effect on both hero buttons.
- `data-reveal` fade-up-on-scroll applied to 20 separate elements across the site —
  every section head, every card grid, the FAQ list, the about card, all reveal on
  scroll with a 0.7s transition, delaying the appearance of primary content.
- `data-tilt` + `data-spot` cursor-tracking 3D tilt and spotlight-glow on every
  `.card` in every grid (services, home).
- A rotating conic-gradient "premium" border glow on card hover, and a diagonal
  "shine sweep" animation on the primary button on every hover.
- A fixed, full-viewport SVG film-grain noise texture (`.grain`) over the entire
  site, at all times, on every page.
- Every service/feature is inside an identical card with an identical circular
  gradient-filled icon well (`.card-icon`, `.tech-icon-wrap`, `.feature-num`) —
  the "icon circle beside every feature" pattern by name.
- Every button is a full pill (`border-radius: 9999px`).

## Strongest elements worth preserving

- **The sidebar app-shell layout** (`.sidebar` / `.topbar` / `.app-footer`) is a
  genuinely distinctive, non-template choice for a marketing site and works well
  responsively (off-canvas drawer below 980px). Keep it.
- **The tech-stack filter** (`.tech-tabs` / `.tech-grid`, `script.js` lines
  350–374) is a real, useful interaction — filtering by category is a legitimate
  use of client-side JS, not decoration.
- **The "Why" timeline** with scroll-driven fill progress is a good use of motion:
  it communicates progress through a real 5-phase process, which is exactly the
  kind of motion the brief says to keep.
- **The FAQ accordion** using native `<details>/<summary>` — zero-JS, accessible
  by default, correctly wired to matching `FAQPage` JSON-LD.
- **The contact form** already has real states: inline field-level validation with
  `aria-describedby`-linked error text, a disabled/"Sending…" submit state, and a
  graceful "form isn't connected yet" fallback message. This is above the bar for
  a template site and should be preserved, not rebuilt.
- **The honest copy voice** ("early-stage studio," "no fixed price list," "real
  case studies should be specific") is unusual and good — the opposite of generic
  SaaS language. The visual design should earn the same trust the copy already has.

## The main "vibecoded" signals

1. **Three unreconciled accent colors.** The `:root` token `--accent: #396287`
   (steel blue) is used only 9 times. The *old* dark-theme accent,
   `rgb(11, 143, 220)` (a brighter cyan-blue), is hardcoded 36 times across glows,
   borders, and shadows. A third color, `#17d6c6` / `rgb(157, 223, 255)` (teal),
   appears 12 times, including baked into two inline SVG checkmark data-URIs. This
   is a straightforward, fixable "too many accent colors" bug, not a design choice.
2. **44 of 184 CSS classes (~24%) are orphaned** — never referenced by any
   generated HTML: an entire unused top-nav header system (`.site-header`,
   `.nav-links`, `.nav-inner`, `.nav-actions`, `.nav-cta`), an unused "trusted by"
   logo marquee (`.trusted`, `.marquee-track`) left over from before the fake-logo
   section was removed, unused testimonial styles (`.quote`, `.avatar`), unused
   fake-case-study metric styles (`.case-img-placeholder`, `.metric-val`), an
   unused cost-estimate summary box (`.summary-price-box`) left over from a removed
   pricing feature, and a whole unused legacy footer (`.site-footer`,
   `.footer-inner`, `.footer-col`) superseded by `.app-footer`.
3. **A WCAG contrast failure inherited from the dark theme**: `.field-error` text
   is `#ff9aa6` (a light pink tuned for a dark background) on a white surface —
   this fails 4.5:1 AA badly and is the exact text a user needs to read to fix a
   form mistake.
4. **`alert()` for planner validation** (`script.js` `validateStep()`) — the
   project planner's step validation uses a native `alert()` dialog, which is
   jarring, blocks the JS thread, and is inconsistent with the real inline
   `field-error` pattern already used two steps later in the same form.
5. Generic decoration list (see above): aurora, glow blobs, glass cards, orbit
   rings, particle sphere, gradient shimmer text, cursor-follow spotlight,
   magnetic buttons, scroll-reveal-everything, cursor-tilt-everywhere, rotating
   card-hover glow, button shine-sweep, full-viewport grain texture, pill buttons,
   icon-circle-per-feature.

## Accessibility problems

- Form error text contrast failure (above).
- `alert()` blocking dialog instead of accessible inline messaging (above).
- Mobile sidebar drawer (`#sidebar.open`) has no focus management: opening it does
  not move focus into it, and closing it does not return focus to the toggle
  button; a keyboard user can currently tab out of the open drawer into content
  hidden behind the backdrop.
- `data-tilt`/`data-spot` and magnetic buttons only run on `pointermove`, so they
  have no keyboard equivalent — not a violation by themselves, but they're
  decoration with no accessible purpose, which is the brief's actual objection.

## Responsive-layout problems

- The hero's 3D scene is simply `display: none` below 540px rather than replaced
  with anything — mobile users get a shorter hero with an abrupt content gap.
- `.tech-tab` buttons (10px vertical padding) are borderline under the 44px touch
  target guidance on mobile.

## Interaction-state problems

- Planner "Next" validation uses `alert()` instead of inline state (above).
- Card hover states (lift + rotating border) fire identically for every card
  regardless of whether the card is meant to be interactive or just informational.

## Performance / layout-stability risks

- `orb-canvas` runs a `requestAnimationFrame` loop continuously while in view
  (320-point 3D projection recomputed every frame) purely for hero decoration —
  unnecessary main-thread work with no product-communication value.
- `pointermove` listeners on `window` (tilt, magnetic, spotlight, 3D scene) run on
  every pointer move across the whole page, not just within the hero.
- Google Fonts request pulls 3 families / many weights
  (`Plus+Jakarta+Sans:...800;900` + `Inter:...700` + `Public+Sans:...700`) —
  more weights than the type system actually differentiates on screen.

## Design-system / maintainability problems

- No spacing scale — section padding is hardcoded per breakpoint in at least 4
  separate places (`104px`, `116px`, `76px`, `64px`, `58px`) rather than driven by
  a token.
- Three separate ad-hoc heading-size `clamp()` calls (`hero h1`, `section-head h2`,
  `page-head h1`) that are all serving the same "page-level display heading" role
  with slightly different values — should be one token, reused.
- Icon-circle-per-feature pattern (`.card-icon`, `.tech-icon-wrap`, `.feature-num`)
  is defined three times with near-identical CSS instead of once.

## Implementation plan

**Critical**
- Fix the form-error contrast failure.
- Replace `alert()`-based planner validation with inline, accessible messaging.
- Add focus management to the mobile sidebar drawer.
- Consolidate the three accent colors into the one real token everywhere.

**High-impact**
- Remove the decorative hero scene (aurora, glow, grid, spotlight, 3D glass scene,
  particle canvas, magnetic buttons, gradient shimmer text) and replace the hero's
  visual half with real, on-brand content: a compact, priority-ordered service
  list that leads with what the business actually leads with (AI agents/multi-agent
  first), instead of a decorative graphic with no information value.
- Remove scroll-reveal-on-everything and cursor-tilt-on-every-card.
- Replace the icon-circle-per-card treatment with a plainer, denser presentation;
  restructure the repeated 3-column service grid (used identically on both the
  home page and the services page) into a single ranked list component that
  reflects the real priority order, reused in both places.
- Remove the `.grain` noise overlay, card hover rotating-border glow, and button
  shine-sweep.
- Delete the 44 orphaned CSS classes / ~300 lines of dead styles.
- Move pill (`9999px`) buttons to a restrained control radius.

**Optional / deferred**
- Trim Google Fonts weights to only what's used.
- Consolidate the three near-duplicate heading `clamp()` scales into shared tokens
  (partially covered by the above; full token pass deferred if time-boxed).

## Completed work

All items in the Critical and High-impact tiers above were implemented directly
(not just recommended). Summary:

- **Accent color consolidated.** `--accent: #396287` is now the only accent color
  referenced anywhere in `styles.css` (via `--accent-rgb` for opacity variants).
  The 36 hardcoded `rgb(11, 143, 220)` references and 12 `#17d6c6`/light-cyan
  references are gone — they were deleted along with the decorative blocks that
  used them (aurora, glow, glass scene, sparks, card border-glow).
- **Decorative layer removed**: aurora animation, hero glow/grid/spotlight, the
  3D glass-card scene, the canvas particle sphere and its `requestAnimationFrame`
  loop, gradient shimmer text, cursor-following magnetic buttons, cursor tilt on
  every card, the rotating card-hover border, the button shine-sweep, and the
  full-viewport film-grain overlay.
- **`data-reveal` scroll-fade removed from all 20 elements** — content is visible
  immediately; a global `prefers-reduced-motion` rule now cuts all remaining
  animation/transition durations to near-zero for users who ask for it.
- **44 orphaned CSS classes removed** along with the decorative rules above;
  `styles.css` went from 1249 to 682 lines. A follow-up automated check (classes
  referenced in HTML/JS vs. defined in CSS, and vice versa) confirms no genuine
  orphans remain on either side.
- **Icon-circle-per-feature pattern removed.** The repeated 3-column service card
  grid (identical on the home and services pages) was replaced with one ranked
  list component (`.service-list` / `.service-row`), reused in both places, with
  a visible rank number instead of a decorative icon well. `SERVICES` is now
  ordered by actual business priority (AI agents/multi-agent first, then
  internal tooling, then web development) — this order was previously arbitrary
  and buried "AI Automation" third out of six.
- **Hero rebuilt** around real content: a "what we lead with" priority panel
  (the top 3 services, ranked) replaces the decorative 3D scene. The headline and
  subhead were rewritten from generic ("technology that moves your business
  forward") to specific (states the actual priority order and "scoped clearly
  before any estimate," both facts pulled from existing site copy/business
  priorities, not invented).
- **CTA band redesigned**: no glow blobs, no big rounded pill container, no
  redundant eyebrow label — a plain, left-aligned section with one button.
- **WCAG contrast bug fixed**: `.field-error` text moved from `#ff9aa6` on white
  (verified ~2:1, failing) to `#b3273f` (verified 6.4:1, passes AA). Consolidated
  three inconsistent reds and a stray blue "success" color into two real semantic
  tokens (`--color-error`/`--color-success` + `-bg` pairs), each verified ≥4.5:1
  against its background.
- **`alert()` replaced** with an inline, `role="alert"` error region
  (`#plannerStepError`) matching the form's existing inline-error pattern;
  verified via Playwright that no native dialog fires and the message reads
  correctly.
- **Mobile sidebar drawer focus management added**: opening moves focus into the
  drawer, `Tab`/`Shift+Tab` are trapped inside it while open, `Escape` closes it
  and returns focus to the toggle button — verified with a keyboard-only
  Playwright script (no mouse events).
- **Buttons moved off full pill radius** (`9999px` → `10px`) for primary/ghost
  CTAs; small tag/filter pills (`.case-tag`, `.tech-tab`) were kept as pills,
  since that's a reasonable, restrained convention for tags/filters rather than
  a "generic SaaS" tell.
- **Dead JS removed**: the count-up stats block (its `data-count` markup no
  longer existed), the scroll-reveal `IntersectionObserver`-style logic, the
  tilt/spotlight/magnetic/3D-particle-sphere pointer handlers. `script.js` went
  from 561 to ~350 lines.

### A real bug caught during verification (not present before this work)

While screenshotting responsive breakpoints, the initial rewrite produced a
genuine horizontal-overflow bug below 980px: the topbar logo `<img>` carries
`width="1500" height="390"` HTML attributes, and the new CSS only constrained
`height`, leaving the browser to resolve `width` from the attribute (1500px)
instead of scaling proportionally, since its `inline-flex` parent has no
explicit width for `max-width: 100%` to resolve against. Fixed by adding
`width: auto` alongside `height` on both logo rules. Confirmed fixed at all six
required breakpoints (320/375/768/1024/1280/1440 — zero overflow at any of them)
via a Playwright script measuring `document.documentElement.scrollWidth`
against the viewport width.

## Deferred / not done

- Trimming Google Fonts weights to only what's actually used on screen (3
  families, several unused weights) — left as-is; a real but lower-priority
  performance item, listed as "optional" in the original plan.
- Full token-name normalization (e.g. renaming `--muted`/`--text-2`, which are
  currently two names for the identical color) was intentionally **not** done —
  the names are already reasonably semantic, and a repo-wide rename would be
  unrelated large-scale refactoring for no visible benefit, which the brief
  explicitly asks to avoid.
- Real case-study content on the Work page — the page already, correctly,
  refuses to show fabricated client results; that's a content decision for the
  site owner, not something a design pass should manufacture.

## Accessibility findings (final)

Fixed: form-error contrast, `alert()` dialog, sidebar drawer focus trap/return.
Verified passing via automated checks: WCAG AA contrast on all primary text/
background/status-color pairs (4.74:1–12.76:1), keyboard-only drawer operation,
zero console/JS errors across all 9 pages. Not independently re-verified in this
pass (inherited from before, appeared correct on inspection): heading order,
landmark structure (`header`/`nav`/`main`/`aside`/`footer`), form label/
`aria-describedby` wiring — these were already correct in the prior version and
were not touched.

## Performance findings (final)

Removed the continuous `requestAnimationFrame` particle-sphere loop and four
separate `window` `pointermove` listeners that ran on every pointer move across
the whole page regardless of scroll position — all previously running purely
for hero decoration with no product-communication value. Not addressed (listed
above as deferred): Google Fonts payload size.

## Verification results

- `node build.mjs`: succeeds, regenerates all 9 pages, no errors.
- `node --check` on `build.mjs` and `script.js`: syntax OK.
- No lint/typecheck/test suite exists in this repo to run as a baseline (no
  `package.json`); this is stated accurately rather than fabricated.
- Playwright, headless Chromium, against a local static server:
  - All 9 routes return correct HTTP status (200 for real pages, 404 for a
    missing route), zero JavaScript console errors or page errors on any page
    (the only console entry anywhere is a Google Fonts request intentionally
    blocked in the test harness, since this sandbox's network policy blocks
    `fonts.googleapis.com` — not a site defect).
  - Zero horizontal overflow at 320/375/768/1024/1280/1440px (caught and fixed
    a real regression at these widths — see above).
  - Tech-stack filter, FAQ accordion, planner step validation + advancement,
    and mobile sidebar drawer (keyboard-only, including focus trap and focus
    return) all independently verified interactively, not just visually.
  - Contrast-checked the concrete color pairs used for body text, muted text,
    accent text, and both status colors against their real backgrounds — all
    pass WCAG AA.
- Not run/verifiable in this sandbox: real Google Fonts loading (network policy
  blocks the domain — verified the block is specifically that domain, not a
  site bug, via the proxy's own status endpoint), Lighthouse/Core Web Vitals
  measurement (no such tool available here), and manual testing in an actual
  screen reader (no screen reader available in this environment — the
  Playwright checks above verify keyboard operability and DOM/ARIA state, not
  actual screen-reader announcement).
